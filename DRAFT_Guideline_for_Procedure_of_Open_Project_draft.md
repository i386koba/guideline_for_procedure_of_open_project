#プロジェクト成果のオープン化ガイドライン


##概要、構成

プロジェクトの成果をオープン化するプロセスは、プロジェクトにおける他の制作プロセスと一体不可分なものです。メディアアート/インタラクションデザインの制作プロセス(研究・開発プロセス)の時期は大きく3つに分けることができます。このオープン化プロセスのガイドラインもそれぞれの時期(pre-production、 production、 post-produciton)にあわせ、3章で構成されています。



>3つの時期(pre-production、production、post-produciton)は、映画の制作手法から引用しています。メディアアート/インタラクションデザインにおける制作では、各章に示された作業は重なり合いながら進むことが多く、この分類は意味をなさないように思えるかもしれません。しかし、これは手続の順序を示すものでもあり、前後関係を把握することは、無理のないプロジェクト進行に繋がります。たとえば、ウェブサイト構築に必要な費用や工期、担当者を事前に把握・設定した方が(企画書に記載した方が)、そうでない場合に比べて構築の実作業を進めやすくなります。

##1.Pre-Production


プロジェクトにおけるオープン化のアウトラインを決めるため、オープン化についての主要な要素を事前に検討します。ここではそれぞれの要素を5W1Hに分類しています。


>それぞれの要素は(特にメディアアート/インタラクションデザイン分野では)、制作の進行に応じて変化する場合があります。しかし、事前にオープン化のアウトラインを示し、プロジェクトメンバーなどの関係者とコンセンサスをとっておけば、制作中の変更に対応も容易になります。外部から参加するコラボレータがいる場合は、その効果はより高くなるでしょう。明示的にコンセンサスをとるため、ここで定めた内容は、企画書に(コラボレーションである場合は契約書にも)明示すべきです。



[YCAM] アウトラインと企画書、契約書

本章で設定した事項をなるべく具体的に企画書に記述します。共同制作(共同研究開発)プロジェクトの場合は、これらの事項を契約書に盛り込みます。企画段階で具体的に決定できない要素がある場合は、その要素について一定の方向性を示すべきです。

共同制作(共同研究開発)プロジェクトの場合は、コラボレータ(アーティスト、共同研究者など)とのコンセンサスをとり、オープン化を行うことの同意、権利の帰趨、オープン化の実作業の分担について、契約書に明示し契約する必要があります。複数の権利処理、作業のわりあてが必要になりますが、GRPContractFormを利用することで解決することができます。


>ここで定めたアウトラインは、productionの中心的な作業にも影響します。たとえば、利用するソースコードやコンテンツが変わり得ます、つまり、第三者が成果を商用利用できるよう公開することを前提としているなら、商用利用を禁止しているソースコードやコンテンツを利用しないことになります。



###why: なぜオープン化するのか

なぜプロジェクトの成果をオープン化するのか、どういった効果を期待できるのかを検討し、オープン化の目的を設定します。
以降のさまざまな要素、プロセスに影響を与える重要な要素です



[YCAM] オープン化の目的

YCAMは公共のアートセンターであり、生み出された成果を市民が広く利用できるようオープン化することは、そのミッションと合致します。さらにオープン化に伴う波及を通じてあたらな創作を促し、YCAMのプレゼンスを高め得ます。必然的にアーカイブを作成することになるので、技術の蓄積も行うことができます。よって、一般論として、YCAMのプロジェクトの成果をオープン化することは、無理がないと考えられます。さらに、プロジェクトごとのオープン化の目的をより具体的に設定すべきでしょう。


>このテキストはオープン化することを啓蒙したり"布教"したりすることを目的とはしていません(以前はオープンソースは一種の宗教である、と揶揄されたそうです。)。もしあなたが関わるプロジェクトにおいて、オープン化の効果が全くない、CPが悪すぎるのであれば、オープン化を見送ることも検討すべきです。



>オープン化には、ウェブサイトの構築やドキュメントの作成、特許制度における新規性の喪失など、一定のコスト(労力、機会損失)が必要になります。こうした"コスト"に対する対価である"ベネフィット"の説明としては、以前は、"コミュニティへの貢献の喜び"や"コミュニティからの評価"があげられることが多くありました。

>一方で、近年ではオープン化によって収益を増やすことができる、というビジネスモデルが提案/実践されるようになってきました。(見たことがないなら、ちょっと検索してみて下さい。)また、公共性のある組織(大学、研究機関、アートセンター)にとっては、そこで生み出された成果を市民が広く利用できるようオープン化することは、そのミッションと対立するものではなく、自然なことです。さらにオープン化に伴う波及を通じて、その組織のプレゼンスを高めるツールとして用いることもできるでしょう。(あなたのラボの研究開発の成果が他の大学のラボで用いられて派生研究が生まれたり、あなたがつくった映像インタラクションシステムをだれかがターミナル駅のガイドに用いたり。)

>もちろん、誰かが、あなたの成果をまるで自分の成果であるかのように用いてしまう、という"リスク"も"ゼロ"ではないかもしれません。成果を独占排他的に運用(exclusiveに運用)することで、確実にベネフィットを増やすことができるなら、そちらを優先すべきかもしれません。逆に、オープン化しないことで(=お蔵入りしてしまうことで)、今後誰にも用いられないという"リスク"も"ゼロ"ではないかもしれません。(exclusiveな運用とopenな運用は相反するものではなく、組み合わせて用いることができます。post-productionを参照ください。)

>もしあなたが、自身のプロジェクトをオープン化した経験がなく、プロジェクトのオープン化を行うべきか否かを迷っているのなら、まず(このガイドラインに従って)ひとつのプロジェクトをオープン化してみて、そのワークフローを経験することを強く勧めます。(小さなプロジェクトなら意外とすぐにできてしまいます。)

###what: 何をオープン化するのか

オープン化の対象の大枠(プロジェクトのコアになる技術なのか、作品自体なのかetc.)をまず決め、どの技術、コンテンツをオープン化するのか、なるべく具体的に想定します。

```
コラボレーションの場合は、コラボレータによって、公開対象が大きく変わる傾向があります。

コラボレータがオープンカルチャーになじみのある場合と、伝統的なアートシーンで活躍している場合では、コラボレータにオープン化を提案したときの受け入れられ方は異なりがちです。後者の場合でコラボレータがオープン化に否定的な様子であっても、オープン化に価値があると考えられる場合は、きっちりとした趣旨説明を行い、オープン化に協力を求めるべきです。
```

###who: 誰がオープン化を担当するのか

誰が何のオープン化の実作業を担当するのか、事前に設定します。

```

プロジェクトがコラボレーションである場合は、特に明文化し契約書に盛り込むことは重要で、事前に定めておけば、スケジューリング、プロセスの進展がスムースになります。

>[YCAM] オープン化の担当者
>
